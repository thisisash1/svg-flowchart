# Figma-Ready SVG Flowchart Skill

피그마에 바로 import할 수 있는 고품질 SVG 플로우차트를 생성하는 스킬이다.

**실행 순서 (절대 변경 불가)**:
1. Phase 0 — 아키텍처 완성도 검증 (논리 구조 확정)
2. Phase 0.5 — 좌표 레지스트리 작성 + 충돌 검증 (**SVG 생성 전 필수**)
3. Phase 1 — SVG 제작 (레지스트리 좌표만 사용)

**절대 하지 말 것**: Phase 0.5 없이 바로 SVG 작성 / 레지스트리 없이 좌표를 임의로 입력 / 충돌 검증 미완료 상태에서 진행

---

## Phase 0: 아키텍처 완성도 검증

> **원칙**: 플로우에 빈틈이 없을 때까지 집요하게 질문한다. 사용자가 명시적으로 확인해야만 SVG 생성으로 넘어간다.

### 0-1. 다이어그램 유형 감지

| 유형 | 키워드 |
|------|--------|
| **서비스 플로우** | 사용자 여정, 화면 전환, UX 흐름, 회원가입, 결제, 로그인 |
| **시스템 아키텍처** | 컴포넌트, 서버, DB, API, 인프라, 백엔드, 마이크로서비스 |
| **유저 저니** | 감정, 경험, 터치포인트, 페인포인트, 여정 지도 |
| **의사결정 트리** | 분기, 조건, 판단, if/else, 케이스 |
| **API/데이터 플로우** | 데이터 이동, 요청/응답, 파이프라인, ETL |

### 0-2. Round 1 — 기반 질문

사용자가 제공하지 않은 항목에 대해서만 질문한다:

1. 이 서비스/기능의 **목적**과 **해결하는 문제**는 무엇인가요?
2. **주요 액터** 목록 (사용자 유형, 내부 시스템, 외부 서비스)
3. **시작 트리거**: 무엇이 이 플로우를 시작시키나요?
4. **종료점**: 정상 종료, 오류 종료, 이탈 종료 각각 어떤 상태인가요?
5. **핵심 단계**를 순서대로 나열해주세요.

### 0-3. Round 2 — 유형별 심화 질문

#### 서비스 플로우
- 각 단계에서 사용자 행동 / 시스템 처리 내용
- 인증/로그인이 필요한 단계
- 단계별 필요 데이터/입력값
- 사용자 피드백(메시지, 화면, 알림) 단계

#### 시스템 아키텍처
- 컴포넌트 전체 목록 + 역할
- 컴포넌트 간 데이터 흐름 방향 (A→B→C)
- 외부 API/서드파티 의존성
- 데이터 저장소 종류 + 위치
- 동기 vs 비동기 혼합 여부

#### 유저 저니
- 각 터치포인트에서 사용자 감정/기대
- 페인포인트 발생 구간
- 기대 vs 실제 결과 불일치 구간
- 의사결정 포인트

#### 의사결정 트리
- 분기 조건 목록 + 조건값
- 각 분기 양쪽(예/아니오) 경로
- 중첩 분기 깊이
- 리프 노드(최종 결과) 목록

#### API/데이터 플로우
- 출처(Source)와 목적지(Destination)
- 중간 변환/처리 단계
- 에러 데이터 처리
- 처리 순서(순차/병렬/조건부)

### 0-4. Round 3 — 갭 탐지

아래 12개 항목을 하나씩 확인한다. "모르겠다" 답변에는 선택지 2~3개를 제시하여 결정을 유도한다.

| # | 항목 | 확인 질문 |
|---|------|----------|
| 1 | **오류/실패 케이스** | 각 단계에서 실패하면 어디로 가나요? |
| 2 | **인증 실패** | 권한 없는 사용자가 접근하면 어디로 리다이렉트? |
| 3 | **타임아웃** | 시스템 응답이 지연/중단되면 어떻게 처리? |
| 4 | **취소/중도 이탈** | 중간에 나가거나 취소하면? 데이터 유지? |
| 5 | **재시도 로직** | 실패 시 재시도 가능? 횟수/시간 제한? |
| 6 | **비인증 경로** | 비로그인/비회원 접근 시 처리? |
| 7 | **외부 시스템 장애** | 외부 API 다운 시 fallback? |
| 8 | **유효성 실패** | 잘못된 입력값 처리? |
| 9 | **병렬 처리** | 동시 실행 단계가 있나요? |
| 10 | **조건부 분기** | 어떤 조건에 따라 경로가 달라지나요? |
| 11 | **알림/이벤트 발송** | 알림(이메일, SMS, 푸시)이 발송되는 단계? |
| 12 | **권한 분기** | 역할(admin/user/guest)별 경로 차이? |

**재질문 규칙**:
- "모르겠다" → 선택지 A/B/C 제시
- 해당 없다 → "정말 이 케이스는 발생하지 않는 게 맞나요?" 한 번 더 확인
- **탈출 조건**: 같은 항목 "모르겠다" 3회 반복 → "기본값으로 처리하고 진행할까요?" 후 진행

### 0-5. 완성도 체크리스트

모두 ✅ 상태여야 다음 단계로 진행 가능하다:

```
[ ] 시작 트리거 1개 확정
[ ] 정상 종료점 확정
[ ] 오류/실패 종료점 확정
[ ] 이탈/취소 종료점 확정 (또는 '없음' 명시)
[ ] 모든 분기의 양쪽 경로 정의 완료
[ ] 오류/실패 처리 경로 최소 1개 정의
[ ] 모든 액터 식별 완료
[ ] 외부 의존성 파악 완료 (없으면 '없음')
[ ] 인증/권한 분기 처리 방식 확정
[ ] 엣지 케이스 (타임아웃, 재시도, 이탈) 검토 완료
```

### 0-6. 플로우 블루프린트 제시

```
📋 플로우 블루프린트: [서비스/기능명]
유형: [...] / 액터: [...]

[START] 시작 트리거

MAIN FLOW:
  Step 1. [색상] 액션/처리
    ├─ 성공 → Step 2
    └─ 실패 → E1
  ...

[END - 정상] / [END - 오류] / [END - 이탈]

EXCEPTION CASES:
  E1. [예외명] — 조건 → 처리 → 복귀/종료

총 노드 수: 약 N개 / 예외 케이스: N개
예상 캔버스 높이: NNNNpx
처리 방식: {N ≤ 20 → "통합 레지스트리" / N ≥ 21 → "레이어별 분할 처리 (자동)"}

이 구조로 SVG를 생성할까요?
```

### 0-7. 생성 게이트

명시적 확인 표현이 있어야만 Phase 0.5로 진입:
- 확인: "예", "네", "응", "진행해", "맞아", "이걸로 해줘", "생성해줘", "OK", "ok", "ㅇㅋ", "좋아", "go", "해줘"
- 수정 요청 → 해당 부분 수정 후 블루프린트 재제시
- 불확실한 답변("글쎄...", "잘 모르겠는데...") → "확인하고 진행할까요, 수정할 부분이 있나요?" 재확인

## Phase 0.5: 좌표 레지스트리 + 충돌 검증

> **절대 원칙**: SVG 코드를 한 줄도 작성하기 전에 이 단계를 완전히 완료해야 한다.

### 0.5-0. 복잡도 게이트 (필수 첫 단계)

Phase 0-6 블루프린트에서 확정된 **총 노드 수**를 기준으로 처리 방식을 결정한다:

| 노드 수 | 처리 방식 |
|---------|----------|
| **≤ 20노드** | 통합 레지스트리 방식 → 0.5-1~0.5-4 순서대로 진행 |
| **≥ 21노드** | 레이어별 분할 처리 → 0.5-L 섹션으로 진입 |

> **이유**: 21노드 이상에서 전체 좌표 레지스트리를 한 번에 계산하면 모델의 thinking token이 10만개+를 소모하다가 API 타임아웃으로 세션이 죽는다. 레이어를 쪼개면 각 단위가 4~8노드로 줄어들어 안전하게 처리된다.

---

### 0.5-L. 레이어별 분할 처리 (21노드 이상 전용)

> 전체 레지스트리를 절대 한 번에 작성하지 않는다. **레이어 하나씩 순환**한다.

**Step 1: 레이어 목록 확정**

블루프린트 구조를 레이어(Layer)로 분리한다. 각 레이어는 최대 8노드.

예시 (SNS 아키텍처):
```
Layer 1: Client        (Mobile App, Web App)                         — 2노드
Layer 2: Edge          (CDN, Load Balancer)                          — 2노드
Layer 3: Gateway+Auth  (API Gateway, Auth Service, OAuth Provider)   — 3노드
Layer 4: Services-A    (User, Post, Media, Feed, Story, Reels, DM)  — 7노드
Layer 5: Services-B    (Comment, Like, Search, Explore, Notif, Mod, Analytics) — 7노드
Layer 6: Event Bus     (Kafka + 이벤트 레이블)                       — 1노드
Layer 7: Storage       (User DB, Post DB, S3, Redis, ES, Neo4j)     — 6노드
Layer 8: External      (Push Gateway, AI/ML Pipeline)               — 2노드
```

**Step 2: SVG 파일 초기화 (한 번만)**

```bash
mkdir -p outputs
cat > outputs/{filename}.svg << 'INITEOF'
<svg xmlns="http://www.w3.org/2000/svg"
  viewBox="0 0 {WIDTH} {HEIGHT}" width="{WIDTH}" height="{HEIGHT}"
  font-family="'Pretendard', 'Apple SD Gothic Neo', 'Noto Sans KR', sans-serif">
<defs>
  <!-- 6종 마커 + grid 패턴 -->
  ...
</defs>
<!-- 배경 -->
<rect width="{WIDTH}" height="{HEIGHT}" fill="#F8FAFC"/>
<rect width="{WIDTH}" height="{HEIGHT}" fill="url(#grid)"/>
<!-- 타이틀 헤더 -->
...
<!-- 범례 바 -->
...
INITEOF
echo "SVG 초기화: outputs/{filename}.svg"
```

**Step 3: 각 레이어 순환 처리**

레이어마다 아래 3단계를 반복한다. 다음 레이어로 넘어가기 전에 현재 레이어를 완전히 완료한다.

```
[레이어 N 처리]
  a) 미니 레지스트리 작성 (해당 레이어 노드만, 최대 8행)
  b) 레이어 내 화살표 경로 계획
  c) Rule 1-5 검증 (해당 레이어 범위만)
  d) SVG 섹션 생성 → cat >> 로 파일에 누적
     cat >> outputs/{filename}.svg << 'LAYEREOF'
     <!-- Layer N: {레이어명} -->
     ...노드 SVG...
     ...화살표 SVG...
     LAYEREOF
     echo "Layer N 추가 완료"
```

**Step 4: 레이어 간 연결 화살표 처리**

모든 레이어 완료 후, 레이어를 넘나드는 연결 화살표만 모아서 한 번에 처리한다.

**Step 5: SVG 마무리**

```bash
cat >> outputs/{filename}.svg << 'ENDEOF'
<!-- 푸터 -->
...
</svg>
ENDEOF
echo "SVG 완성: outputs/{filename}.svg ($(wc -c < outputs/{filename}.svg) bytes)"
```

> 0.5-L 완료 후 Phase 1의 SVG 파일 저장 단계는 건너뛰고, 기획 MD 저장 단계만 실행한다.

### 0.5-1. 레이아웃 방향 + 캔버스 크기 확정 (20노드 이하 전용)

`references/layout-guide.md` 참조하여 세로/가로/스윔레인/혼합 중 선택.

### 0.5-2. 노드 레지스트리 작성

플로우의 **모든 노드**를 테이블로 나열한다. SVG 좌표는 이 테이블에서만 가져온다.

```
| ID | 유형 | x | y | w | h | cx | cy | entry_y | exit_y | exit_x_L | exit_x_R |
|----|------|---|---|---|---|----|-----|---------|--------|----------|---------|
| title  | header  | 60  | 40  | 1680 | 72 | 900 | 76  | —   | —   | —   | —    |
| start  | pill    | 700 | 208 | 400  | 48 | 900 | 232 | 208 | 256 | —   | —    |
| step1  | rect    | 580 | 292 | 640  | 80 | 900 | 332 | 292 | 372 | 580 | 1220 |
| d1     | diamond | —   | —   | —    | —  | 900 | 440 | 404 | 476 | 780 | 1020 |
```

규칙:
- `entry_y` = 노드 상단 y / `exit_y` = 노드 하단 y
- `exit_x_L` = 노드 왼쪽 x / `exit_x_R` = 노드 오른쪽 x + width
- 수직 간격은 반드시 계산으로 결정, 추정 금지

### 0.5-3. 화살표 경로 계획 테이블

```
| 화살표 ID | 출발 노드.포인트 | 도착 노드.포인트 | 경로 유형 | 경유점 | 마커 |
|-----------|----------------|----------------|---------|--------|------|
| a1 | start.exit_y(900,256) | step1.entry_y(900,292) | 직선↓ | — | ah-slate |
| a2 | d1.exit_x_R(1020,440) | end-err.entry(우측) | L형 우회 | (1300,440)→(1300,N) | ah-red |
```

경로 유형별 패턴:
- **직선↓**: `<line x1="{cx}" y1="{exit_y}" x2="{cx}" y2="{next.entry_y}" .../>`
- **L형 우회**: `<path d="M {x} {y} L {wx} {y} L {wx} {ty} L {tx} {ty}" fill="none" .../>`
- **꺾인 점선**: L형 + `stroke-dasharray="5,3"`

### 0.5-4. 충돌 검증 (5가지 규칙, 모두 PASS해야 Phase 1 진입 가능)

**Rule 1**: 인접 노드 `exit_y ≤ next.entry_y` (여백 ≥ 20px) / 위반 → y 값 재계산

**Rule 2**: 화살표 시작/끝 좌표가 노드 경계와 ±5px 이내 / 위반 → 레지스트리 값으로 수정

**Rule 3**: 우회 경유점 wx가 메인 플로우 노드 x 범위(580~1220) 밖 / 위반 → wx를 바깥으로 이동

**Rule 4**: `exception_container.y ≤ 모든 예외 노드 최소y - 20` 및 하단도 동일 / 위반 → 컨테이너 재계산

**Rule 5**: 모든 화살표 길이 > 0 / 모든 화살표에 `marker-end` 존재 / 위반 → 추가 또는 제거

검증 후 결과를 출력한다:
```
Rule 1: ✅  Rule 2: ⚠️ a2 끝점 수정 → ...  Rule 3: ✅  Rule 4: ✅  Rule 5: ✅
→ 1건 자동 수정 완료, 재검증: 전체 PASS
```

**전체 PASS 후 사용자에게 알린다**: "레지스트리 검증 완료. SVG 생성을 시작합니다."

## Phase 1: SVG 제작

> Phase 0.5 전체 PASS 확인 후에만 실행한다.
> **모든 좌표는 Phase 0.5 레지스트리에서 가져온다. 추정하거나 임의로 입력하지 않는다.**

**레퍼런스 파일 (Phase 1 시작 전 Read)**:
- `references/layout-guide.md` — 캔버스 크기, 레이아웃 패턴, 간격 규칙
- `references/component-examples.md` — 컴포넌트별 SVG 코드, 컬러 시스템, 텍스트 규칙

### 제작 절차

1. `references/layout-guide.md` 기준으로 레이아웃 결정
2. `references/component-examples.md` 컴포넌트 패턴 적용
3. Bash로 SVG 파일 생성 (레지스트리 좌표만 사용)
4. 동일 파일명으로 기획 MD 저장

#### SVG 파일 저장

> **21노드 이상 (0.5-L 경로)**: 이 단계는 건너뛴다. SVG는 이미 Phase 0.5-L에서 레이어별로 누적 저장되었다.

> **20노드 이하 (0.5-1~4 경로)**: 아래 명령으로 전체 SVG를 한 번에 저장한다.

```bash
mkdir -p outputs
cat > outputs/{filename}.svg << 'SVGEOF'
<svg xmlns="http://www.w3.org/2000/svg"
  viewBox="0 0 {WIDTH} {HEIGHT}" width="{WIDTH}" height="{HEIGHT}"
  font-family="'Pretendard', 'Apple SD Gothic Neo', 'Noto Sans KR', sans-serif">
  ...
</svg>
SVGEOF
echo "저장 완료: outputs/{filename}.svg ($(wc -c < outputs/{filename}.svg) bytes)"
```

#### 기획 MD 저장

```bash
cat > outputs/{filename}.md << 'MDEOF'
# {다이어그램 제목}
> 생성일: {날짜} · v{N} · `outputs/{filename}.svg`

## 개요
| 항목 | 내용 |
|------|------|
| 유형 | {서비스 플로우 / 시스템 아키텍처 / ...} |
| 레이아웃 | {세로형 / 가로형 / 스윔레인 / 혼합형} |
| 캔버스 | {WIDTH} × {HEIGHT}px |
| 총 노드 수 | 약 {N}개 |
| 예외 케이스 수 | {N}개 |

## 액터
{액터 목록 (역할 포함)}

## 결정 사항
| 항목 | 결정 내용 | 근거 |
|------|-----------|------|
{Phase 0 확정 사항}

## 플로우 블루프린트
{Phase 0-6 블루프린트 전문}

## 메인 플로우 단계
| Step | 유형 | 제목 | 설명 |
|------|------|------|------|

## 예외 처리 케이스
| ID | 트리거 | 처리 흐름 | 종료 유형 |
|----|--------|-----------|-----------|

## 변경 이력
| 날짜 | 버전 | 변경 내용 |
|------|------|-----------|
| {날짜} | v{N} | 최초 생성 |
MDEOF
echo "기획 MD 저장 완료: outputs/{filename}.md"
```

저장 후:
- "SVG: `outputs/{filename}.svg` — Figma에서 드래그&드롭으로 임포트"
- "기획 MD: `outputs/{filename}.md` — 플로우 설계 명세"
